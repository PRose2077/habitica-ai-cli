# Habitica-Forge V2 路线图 (TODO)

## V2 总目标

V1 已经完成了“可用”的 Habitica CLI：能同步、能管理任务、能做 AI 拆解，也有称号、腐烂、风格切换这些基础游戏化能力。

V2 不再只是继续堆功能，而是要解决一个更核心的问题：**现在的游戏化还偏“文案风格化”，不够像一个真正可持续游玩的任务系统。**

这一版的目标，是把游戏化从“写得更酷”升级成“任务天然带有角色扮演、地点、任务类型、阶段推进、奖励反馈”的完整体验。

---

## V2 设计原则

- [x] **游戏化优先于辞藻堆砌**：不是把普通任务随机改几个华丽词，而是让任务具备“身份、地点、目标、进度、奖励”的结构。
- [x] **保留现实语义，不制造理解成本**：所有游戏化标题都必须让用户仍然看得懂原任务，避免为了风格牺牲可执行性。
- [x] **双层信息表达**：用户既可以看到游戏化名称，也可以快速看到现实含义，避免任务“看着热闹、做起来模糊”。
- [x] **统一交互标识，降低输入门槛**：所有命令交互一律使用列表返回的**任务编号**（如 1, 2, 3），彻底废弃手输冗长的 UUID 或短 ID，做到“所见即所填”。
- [x] **统一入口与沉浸体验**：废弃碎片化的智能/游戏化概念，将创建、拆解、包装全部收拢至 `gamify` 命令族，提供无缝体验。
- [x] **风格模板化而不是硬编码**：中世纪、武侠、赛博朋克、科幻等都由统一的 YAML 词典与模板系统驱动。
- [x] **安全可控的视觉资源**：通过内置图片资源词典约束 AI 生成，避免不可控外链导致质量问题。
- [x] **允许破坏式重构，不为 V1 兼容性买单**：V2 的首要目标是建立干净、稳定、可持续演进的任务系统；如果旧结构、旧命令、旧元数据设计会拖累抽象，就直接移除，不额外引入兼容层、双轨逻辑和历史包袱。
- [ ] **先补强任务体验，再扩张系统复杂度**：先把单个任务的游戏化做扎实，再考虑更重的剧情系统。

## 核心问题清单

- [x] **问题 1：V1 的游戏化主要发生在 AI 文案层**，但任务本身缺少稳定的”任务身份结构”。（已通过双名结构与原型系统解决）
- [x] **问题 2：普通 Todo、Daily、Habit 的游戏化策略没有明显区分**，导致不同任务类型的手感不够鲜明。（已通过任务原型专属规则解决）
- [x] **问题 3：AI 介入节点分散**，智能拆解 (`smart`) 和游戏化包装 (`gamify`) 分离，缺乏一站式委托体验。（已通过命令合并解决）
- [x] **问题 4：用户操作成本高**，使用 UUID 或短 ID 难以记忆，极大降低了在 CLI 中快速操作任务的爽快感。（已通过强制使用编号交互解决）
- [x] **问题 5：大任务虽然能拆解，但还不像”传奇任务 / 任务链”**，缺少章节感和推进反馈。（已通过传奇任务系统解决）
- [x] **问题 6：地点、场景、阵营、职业感很弱**，用户难以形成自己的世界观沉浸感。（已通过职业倾向系统和地点命名系统解决）
- [ ] **问题 7：奖励反馈仍偏后置**，完成一个任务时的即时反馈、连续反馈、系列反馈还不够强。

## 阶段一：任务游戏化语义层重构 (Quest Semantics) ✅ 已完成
**目标**：建立 V2 的核心抽象，让”任务为什么像游戏”不再依赖一段随机文案，而是依赖稳定的数据结构。

- [x] **1.1 设计任务双名结构（现实名 + 游戏名）**
	- [x] 明确每个任务至少有两层可读信息：`real_title` 与 `quest_title`。
	- [x] 设计最小存储方案，优先复用 Habitica `notes` 中的隐式元数据，避免引入本地数据库。
	- [x] 约定 V2 元数据标记格式，例如 `<!-- FORGE_META: {...} -->`。
- [x] **1.2 定义任务原型（Quest Archetypes）**
	- [x] 为任务建立有限的语义分类，例如：`清理`、`修复`、`探索`、`制作`、`沟通`、`学习`、`战斗`、`补给`。
	- [x] 让 AI 先判断任务原型，再基于原型生成更稳定的游戏化命名。
	- [x] 为不同原型指定默认动词、名词、语气与奖励倾向。
- [x] **1.3 建立任务类型专属规则**
	- [x] Habit：更强调诱惑/克制、训练/堕落、正负反馈。
	- [x] Daily：更强调巡逻、仪式、例行职责、守护循环。
	- [x] Todo：更强调委托、任务书、战役、支线/主线。
- [x] **1.4 增加游戏化可读性约束**
	- [x] 规定游戏化标题长度上限，避免 CLI 列表难以阅读。
	- [x] 规定每条游戏化标题必须保留一个现实关键词。
	- [x] 当 AI 生成结果过于浮夸或不可执行时，自动回退到保守命名。

## 阶段二：风格系统升级为”世界观词典” (Style Lexicon 2.0) ✅ 已完成
**目标**：让风格不只影响 prompt 语气，而是统一影响任务命名、地点命名、奖励命名和 UI 文案。

- [x] **2.1 重构风格 YAML 结构**
	- [x] 为每个风格增加：任务动词词典、地点词典、称号词典、敌人词典、稀有度词典。
	- [x] 支持 Habit / Daily / Todo 各自独立的命名模板。
	- [x] 保留 `normal` 作为基线风格，确保可退回朴素表达。
- [x] **2.2 引入地点游戏化词典**
	- [x] 支持现实地点到游戏地点的映射，如房间→营地/舱室，厨房→补给站/火坑。
	- [x] 允许通过 tag 或配置声明个人世界观偏好，如 fantasy / cyberpunk / sci-fi。
	- [x] 让地点映射参与任务命名，而不是只作为展示装饰。
- [x] **2.3 引入任务上下文词典**
	- [x] 工作类任务倾向公会、议会、指挥中心等命名。
	- [x] 学习类任务倾向研究、法术、卷轴、档案破解等命名。
	- [x] 家务类任务倾向清剿、净化、维护、补给等命名。
- [x] **2.4 建立风格质量基线**
	- [x] 为每个风格编写 10-20 条标准输入输出样例。
	- [x] 确保同一个任务在不同风格下”语义一致、表达不同”，而不是变成不同任务。
- [x] **2.5 引入图片资源词典 (`images.yaml`)**
	- [x] 新增独立的 `images.yaml`，统一维护可用于任务标题/展示的图片资源。
	- [x] 每个图片条目至少包含：唯一 ID、标题、描述、URL。
	- [x] AI 在生成任务时只读取图片的标题和描述，不直接接触 URL。
	- [x] 程序在最终渲染或写入 Habitica markdown 时，根据 AI 返回的图片 ID 解析对应 URL。
	- [x] 图片资源应支持按风格、任务原型或用途分类，避免 AI 在全量图片中无差别挑选。
	- [x] 默认只启用白名单图片，避免用户随意输入外链导致质量或版权问题。

## 阶段三：从”智能拆解”升级到”任务委托”与统一游戏化入口 (Quest Authoring & Unified Gamification) ✅ 已完成
**目标**：统一 AI 介入入口，将原本重叠的 `smart` 和 `gamify` 命令族合并，让用户创建的不是普通任务，而是一张任务委托单 / 任务书。

- [x] **3.1 合并 `smart` 与 `gamify` 命令**
	- [x] 废弃独立的 `smart` 命名空间，将智能拆解与游戏化概念统一。
	- [x] 移除与 `gamify add` 语义重复的独立 `add` 入口，避免同一能力出现多个入口。
	- [x] 统一约定”创建即游戏化”，不再保留”普通添加”和”游戏化添加”两套并行心智模型。
	- [x] 将原 `smart-add` 升级为 `gamify add`，提供一步到位的智能创建+游戏化体验。
	- [x] 将原 `smart-task` 升级为 `gamify task`（或类似命令），专注对已有任务进行智能拆解与包装。
- [x] **3.2 改造 `gamify add` 输出结构**
	- [x] 除 checklist 外，同时生成：任务原型、游戏名、现实副标题、地点、建议奖励感。
	- [x] 允许 AI 可选返回图片候选 ID，用于后续插入 markdown 图片或图标。
	- [x] 输出结果采用”委托书”样式，而不是普通说明文字。
	- [x] 若是复杂任务，自动标记为”主线任务”或”传奇任务候选”。
- [x] **3.3 改造 `gamify task` 拆解结构**
	- [x] 将 checklist 拆解为”阶段 / 章节 / 小目标”，而不是平铺子任务。
	- [x] 支持为每一步生成简短阶段标题，例如”侦察现场””收集卷轴””提交战报”。
	- [x] 保留 `--keep` 逻辑，但支持对已有 checklist 做”再游戏化整理”。
- [x] **3.4 默认预览与强制提交机制**
	- [x] 游戏化命令（如 `gamify add` / `gamify task`）默认进入**预览与确认流**，先展示生成结果，用户确认后再提交到 Habitica。
	- [x] 提供 `-f` 或 `--force` 参数，允许跳过预览直接应用生成结果。
	- [x] 新增 `forge gamify revert <编号>`，将任务恢复为现实标题优先展示，确保用户随时退出沉浸模式。

## 阶段四：传奇任务与任务链系统 (Legendary Quests) ✅ 已完成
**目标**：解决”大任务只是长 checklist”这个问题，把复杂任务真正做成带推进感的战役。

- [x] **4.1 定义传奇任务判定规则**
	- [x] 基于任务长度、子任务数、时间跨度或用户显式指定判断是否为传奇任务。
	- [x] 为传奇任务增加专属前缀，如 `主线`、`远征`、`战役`、`护送任务`。
- [x] **4.2 设计章节化拆解**
	- [x] 将一个复杂任务拆成 3-5 个章节，每个章节下包含若干 checklist。
	- [x] 在 `notes` 中保留章节摘要，避免 Habitica 只看到零碎子任务。
	- [x] CLI `show` 中优先展示章节进度，而不是只展示 checklist 勾选数。
- [x] **4.3 增加任务链 / 系列任务支持**
	- [x] 同一项目下的任务支持共享系列名，例如”北境迁移计划 I / II / III”。
	- [x] 完成系列时给予额外提示或更高阶称号掉落概率。
	- [x] 支持将多个独立 Todo 编组成一个任务链视图。

## 阶段五：地点、阵营与沉浸式展示 (World Layer) ✅ 已完成
**目标**：让用户不只是”完成任务”，而是在一个有地点感的世界里行动。

- [x] **5.1 地点命名系统**
	- [x] 支持根据 tag、文本关键词或配置识别任务发生地。
	- [x] 在 `list` / `show` 中展示任务所属地点，如”营地””法师塔””指挥甲板”。
	- [x] 地点命名受风格系统统一控制。
- [x] **5.2 阵营 / 职业倾向系统**
	- [x] 基于用户近期任务分布生成职业倾向，例如”抄写员””工程术士””档案猎人”。
	- [x] 职业倾向不替代称号墙，而是作为轻量、动态身份层。
	- [x] Header 中可同时显示：当前佩戴称号 + 当前职业倾向。
- [x] **5.3 图标与视觉标记**
	- [x] 先支持安全、轻量的 emoji 图标前缀。
	- [x] 支持可选 markdown 图片嵌入，但默认关闭，由用户显式开启。
	- [x] 图片能力统一走 `images.yaml` 资源词典，不允许 AI 直接生成任意图片 URL。
	- [x] 图片提示词只暴露标题和描述，URL 注入必须由程序在落地阶段完成。
	- [x] 为图片定义尺寸和使用场景约束，避免任务列表因大图失控膨胀。
	- [x] 为不同任务原型提供统一图标规则，而不是随机插图。

## 阶段六：即时反馈与奖励回路 2.0 (Feedback Loop)
**目标**：增强“完成一项任务时像打了一局游戏”的反馈，而不是只在事后掉一个称号。

- [ ] **6.1 完成反馈文案升级**
	- [ ] `forge done` 后根据任务原型输出不同完成提示，如“据点已清剿”“卷轴已解译”。
	- [ ] 同时保留现实摘要，避免用户不知道自己完成了什么。
- [ ] **6.2 连续行为反馈**
	- [ ] Daily 连击高时显示“巡逻连胜”“守护天数”等轻量反馈。
	- [ ] Habit 连续正向行为可触发“训练阶段提升”提示。
- [ ] **6.3 称号掉落 2.0**
	- [ ] 称号生成不只依赖难度，还参考任务原型、系列任务、地点主题。
	- [ ] 支持“系列称号”进阶，例如学徒→抄写员→大法师。
	- [ ] 降低纯 RNG 感，提升长期可预期成长感。
- [ ] **6.4 腐烂系统与 V2 对齐**
	- [ ] 腐烂文案不再只是变黑暗，而应体现“世界状态恶化 / 任务失守 / 敌人滋生”。
	- [ ] 腐烂等级展示要更可见，但不能污染主列表可读性。

## V2 建议实现顺序

- [x] **P0：先清理 V1 历史包袱，必要时直接破坏式重构**，不为了保留旧入口、旧字段、旧行为继续堆兼容分支，否则 V2 很快会重新退化成难以维护的屎山。
- [x] **P0：先完成任务语义层与双名结构**，这是后续所有游戏化能力的基础。
- [x] **P0：升级风格 YAML 为词典系统**，把风格从”Prompt 语气”提升为”世界观驱动器”。
- [x] **P0：统一 `gamify` 和 `smart` 命令族**，降低用户使用成本和概念理解难度。
- [x] **P1：将原本的智能任务创建与拆解彻底转化为”委托书 / 传奇任务生成”体验**。
- [ ] **P1：重做 list / show 展示层**，让用户日常使用时持续感知到 V2 差异。
- [ ] **P1：把称号、腐烂、完成反馈统一到任务原型体系下。**
- [ ] **P2：再做阵营、职业倾向、图标、任务链等增强体验。**

## V2 验收标准

- [ ] 用户看见一个任务时，**一眼知道这是什么现实任务，也一眼感觉它像一个游戏委托**。
- [ ] 同一个任务在不同风格下，语义保持一致，但世界观表达明显不同。
- [ ] `list` 和 `show` 的体验能稳定体现地点感、任务类型感和阶段推进感。
- [ ] 完成任务时的反馈，不再只有“完成了”，而是有明确的任务结算感。
- [ ] 用户可以随时进入或退出沉浸模式，不会因为游戏化过强而影响实际执行。

---

## 一句话总结

V1 的关键词是：**让任务更聪明。**

V2 的关键词应该是：**让任务真正变成可游玩的委托、战役和日常巡逻。**

补一句执行原则：**宁可明确升级并打破旧设计，也不要为了“看起来平滑”继续背兼容债。**
